# AI Interview Preparation Platform - Technical Architecture & Stack Description

This document provides a comprehensive overview of the technical stack, database schema, AI integration models, and custom algorithms used in both the Frontend (Client) and Backend (Server) of the **AI Interview Preparation Platform**.

---

## 🖥️ Frontend (Client) Architecture

The frontend is a modern, responsive single-page experience designed to act as an interactive IDE and real-time mock interview environment.

### Core Stack & Rationale
1. **Next.js 16 (React 19, App Router)**
   * *Why:* Provides fast, directory-based routing (App Router), optimized server-side rendering (SSR), and Turbopack for rapid local development compilation.
2. **TypeScript (TSX)**
   * *Why:* Enforces strict compile-time type-safety across components, network services, and API payloads.
3. **Tailwind CSS**
   * *Why:* Rapid responsive styling with pre-configured glassmorphism, dark-mode styling, and smooth animations.
4. **Lucide React**
   * *Why:* Clean, modern vector icons for UI navigation and dashboards.

### Custom Algorithms & Hooks
* **Speech-to-Text Grading Loop (`useSpeechRecognition` hook)**:
  * *Where:* Used in the Voice Mock Interview rooms.
  * *How it works:* Integrates with the browser's native **Web Speech API** (`window.SpeechRecognition` / `webkitSpeechRecognition`).
  * *Logic:* Configured with `continuous = true` and `interimResults = true` to capture voice stream in real-time. It accumulates transcribed words dynamically into a single continuous transcript buffer and automatically terminates/flushes on stop or error.

## ⚙️ Backend (Server) Architecture

The backend is a RESTful API designed to parse resume text, manage user sessions, interact with database schemas, and orchestrate calls to the Gemini generative model.

### Core Stack & Rationale
1. **Express.js (Node.js & TypeScript)**
   * *Why:* Minimalist web framework that integrates cleanly with TypeScript. Highly scalable for asynchronous operations like file streams and LLM API calls.
2. **JWT Authentication (`jwt.middleware.ts`)**
   * *Why:* Stateless token authentication. Secures user-specific routes (mock sessions, uploaded resumes) and automatically decodes user IDs.
3. **Multer & File System Management**
   * *Why:* Safely handles multi-part file uploads (PDF resumes), storing them temporarily on disk for processing and automatically deleting them post-analysis to maintain disk cleanliness.
4. **pdf-parse**
   * *Why:* Extracts raw text streams from PDF resume uploads to feed into the Gemini analyzer.

## 🗄️ Database & ORM Architecture

The database keeps track of user growth, test submissions, scoring histories, and customized roadmap timelines.

### SQLite & Prisma ORM
* *Why SQLite:* Uses a local file-based database (`prisma/dev.db`), offering zero-config development and simple setup.
* *Why Prisma:* Simplifies relational schemas, provides type-safe autogenerated databases queries, and makes switching to PostgreSQL for production simple (by changing the `provider` in `schema.prisma`).

### Relational Schema Tables
* **User**: Manages credentials, sign-up times, and relationships to resumes/sessions.
* **Resume**: Stores raw parsed resume text and JSON lists of identified strengths, weaknesses, current skills, and missing skills.
* **Interview & Question & Answer**:
  * Represents mock interviews (10 questions per session).
  * Stores individual voice transcripts (`Answer`) along with scores (1-10) and feedback.
* **LeetCodeSession**: Manages conceptual preparation rounds (saving candidate explanations and AI feedback).
* **CodingAssessment**: Stores the live editor code, language target, Big-O complexities, and code quality reports.
* **Roadmap**: Saves customized 4-week timelines mapping technology gaps to company expectations.

## 🤖 AI Models & Integration Architecture

The intelligence of the platform is powered entirely by Google's state-of-the-art Gemini LLM.

### Model & Configuration
* **AI Model:** `gemini-2.5-flash`
* **SDK:** `@google/generative-ai`
* **API Pattern (Structured JSON Outputs):**
  * Configured with `generationConfig: { responseMimeType: 'application/json' }`.
  * Prompts contain strict JSON schemas. This ensures the model outputs valid JSON strings, which the backend parses directly into database columns.
* **Mock Fallback Pattern:**
  * If no `GEMINI_API_KEY` is provided, the application runs in a robust **Mock Fallback Mode**, returning realistic pre-compiled templates so features remain usable offline/keyless.

## 📊 Core Algorithms & LLM Prompts

### 1. ATS Resume Analyzer (`analyzeResumeText`)
* **Objective:** Scan candidate qualifications.
* **Logic:** Feeds raw text to the model to output a classified breakdown of matching keywords, strengths, and missing skills relevant to modern technical job descriptions.

### 2. Tailored Interview Generator (`generateInterviewQuestions`)
* **Objective:** Produce 10 highly relevant interview questions.
* **Constraint Logic:** 
  * 3 HR/Behavioral questions.
  * 4 Technical concepts (customized to candidate's stack and target role).
  * 3 Project-based questions (extracting details directly from projects listed on their resume).

### 3. Speech/Text Answer Evaluation (`evaluateAnswer`)
* **Objective:** Score candidate responses.
* **Algorithmic rules:**
  * Checks character count. If the response transcript is `< 5` characters or blank, it fails fast and awards a score of `1` with recommendations to expand.
  * Otherwise, prompts Gemini to grade the explanation against a correctness metric, generating specific "Actionable Improvement Tips" (e.g. suggesting structured frameworks like the STAR method).

### 4. DSA Code Assessment & Big-O Estimation (`evaluateCodingSubmission`)
* **Objective:** Compile-less static evaluation of code submissions.
* **Logic:** Evaluates user code lines against:
  * **Syntax & Execution logic** (is it complete?).
  * **Big-O complexity** (evaluates time/space performance mathematically).
  * **Edge case simulation** (detecting nulls, boundary limits, array indexes out of bounds).
  * **Clean-code assessment** (variable naming, modular styling).
